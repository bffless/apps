## How the apps have no server

Neither `apps/handoff` nor `apps/studio` ships a backend. Each app is a static build
deployed to BFFless, and everything the browser calls under `/api/*` is served by a
BFFless **proxy rule set** rather than by a server we run.

A proxy rule set is a named, reusable group of **rules**. Each rule matches a request
by path pattern + HTTP method and decides where it goes:

- **`pipeline` rules** (`proxyType: pipeline`) hand the request to a BFFless
  **pipeline** — a chain of no-code handlers (data create/query/update/delete, file
  upload/serve, presigned URL, AI/Replicate calls, function transforms, response
  shaping). This is how the apps get a backend without writing server code. Most
  rules below are pipeline rules.
- **`external_proxy` rules** (`proxyType: external_proxy`) reverse-proxy to a real
  upstream URL — used here only to forward SuperTokens auth endpoints — which also
  sidesteps CORS and hides the upstream from the client.

A rule set is only active once it is **attached to an alias** (a named pointer to an
immutable deployment) via `proxyRuleSetIds`. When multiple sets are attached their
rules merge in priority order. In this project each app uses a single rule set,
attached to both its production and its preview alias, so production and preview share
identical backend behaviour and differ only in the deployment they point at.

Pipeline rules read and write **pipeline schemas** (BFFless Data Tables) — the
schema-based storage that backs the node tree, jobs, project state, and share links.
Those tables are inventoried at the end.
